Refactor the diagnosis correction strategy to avoid large repair prompts and prevent input loss.

Context:

* diagnosis currently fails fast when validation fails
* a repair strategy is needed
* however, sending the full original diagnosis prompt, full numbered SP, previous output,
  and validation errors together would create oversized repair prompts
* oversized repair prompts risk loss of input and lower correction quality

Goal:
Implement a compact, validation-aware diagnosis repair strategy based on failure classification.

Requirements:

1. Extend diagnosis-validator output so validation results are structured

Suggested shape:

* passed
* errors
* missingSections
* hasLineReferences
* errorTypes

2. Introduce diagnosis failure classification

Suggested categories:

* structural_only
* semantic_or_unknown

3. Structural-only failures include:

* missing required section
* missing line references
* invalid section heading
* empty diagnosis artifact

4. For structural-only failures, allow exactly one repair attempt

Repair prompt must be compact and must include only:

* a short repair instruction
* the previous normalized diagnosis output
* a compact validation summary

Do NOT include:

* the full numbered stored procedure
* the full canonical diagnosis prompt
* unrelated context

5. The repair prompt must explicitly instruct:

* preserve existing valid content
* add missing sections only if needed
* add explicit line references when missing
* do not generate code
* do not invent new business assumptions
* do not redesign the artifact

6. For semantic_or_unknown failures:

* do NOT attempt compact repair
* fail the stage
* persist validation artifact and outputs for inspection

7. Persist validation result in:
   work/<spId>/20-analysis/diagnosis.validation.json

8. Keep the repair strategy deterministic and minimal

Goal:
Prevent oversized repair prompts by introducing failure-aware compact repair,
instead of blindly resending the full diagnosis context.
