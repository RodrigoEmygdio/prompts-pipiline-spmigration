Refactor the diagnosis correction strategy to avoid large repair prompts and prevent input loss.

Context:

* diagnosis currently fails fast when validation fails
* a repair strategy is needed
* however, sending the full original diagnosis prompt, full numbered SP, previous output,
  and validation errors together would create oversized repair prompts
* oversized repair prompts risk loss of input and lower correction quality
* intermediate diagnosis artifacts must be persisted so correction can reuse them directly
  without rebuilding the full diagnosis context

Goal:
Implement a compact, validation-aware diagnosis repair strategy based on failure classification,
while preserving all intermediate diagnosis artifacts.

Requirements:

1. Persist intermediate diagnosis artifacts for every diagnosis execution

Always save:

* work/<spId>/20-analysis/diagnosis.raw.md
* work/<spId>/20-analysis/diagnosis.normalized.md
* work/<spId>/20-analysis/diagnosis.validation.json

If a repair attempt is executed, also save:

* work/<spId>/20-analysis/diagnosis.repair.raw.md
* work/<spId>/20-analysis/diagnosis.repair.normalized.md
* work/<spId>/20-analysis/diagnosis.repair.validation.json

2. Extend diagnosis-validator output so validation results are structured

Suggested shape:

* passed
* errors
* missingSections
* hasLineReferences
* errorTypes

3. Introduce diagnosis failure classification

Suggested categories:

* structural_only
* semantic_or_unknown

4. Structural-only failures include:

* missing required section
* missing line references
* invalid section heading
* empty diagnosis artifact

5. For structural-only failures, allow exactly one repair attempt

The repair flow must use persisted intermediate artifacts:

* diagnosis.normalized.md
* diagnosis.validation.json

The repair prompt must be compact and must include only:

* a short repair instruction
* the previous normalized diagnosis output
* a compact validation summary

Do NOT include:

* the full numbered stored procedure
* the full canonical diagnosis prompt
* unrelated context

6. The repair prompt must explicitly instruct:

* preserve existing valid content
* add missing sections only if needed
* add explicit line references when missing
* do not generate code
* do not invent new business assumptions
* do not redesign the artifact

7. For semantic_or_unknown failures:

* do NOT attempt compact repair
* fail the stage
* persist validation artifact and outputs for inspection

8. The diagnosis.validation.json artifact must be the source of truth for repair decisions

The repair logic must read the persisted validation artifact from disk,
instead of reconstructing validation context in memory from scratch.

9. Keep the repair strategy deterministic and minimal

10. Do NOT redesign:

* context-builder
* provider-adapter
* artifact paths outside diagnosis intermediate artifacts
* stage contracts

Goal:
Prevent oversized repair prompts by introducing failure-aware compact repair,
while preserving raw, normalized, and validation artifacts for auditability and controlled retries.
